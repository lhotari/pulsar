# PIP-485: RoaringBitmap encoding and cursor-ledger compression for acknowledgment state

# Background knowledge

A Pulsar subscription's acknowledgment state is tracked by a **managed cursor**. Two pieces of per-cursor
state can grow large when consumers acknowledge **out of order** (Shared / Key_Shared subscriptions, negative
acknowledgments, delayed delivery, retry/dead-letter flows):

* **Individually deleted message ranges** (`individualDeletedMessages`) — the set of acknowledged messages that
  sit *after* the mark-delete position, i.e. the "ack holes". This is held in memory as a
  `LongPairRangeSet` and, with the default `managedLedgerUnackedRangesOpenCacheSetEnabled=true`, the backing
  set is an `OpenLongPairRangeSet` whose per-ledger bitmap is an `org.roaringbitmap.RoaringBitSet`.
* **Batch deletion indexes** (`batchDeletedIndexes`) — for each partially-acknowledged *batched* entry, a
  `BitSet` of which message indexes inside the batch are acknowledged. Active when
  `acknowledgmentAtBatchIndexLevelEnabled=true` (the default).

This state is persisted on every mark-delete flush. There are two stores:

* **BookKeeper cursor ledger** (`PositionInfo`) — the steady-state target. Each flush appends **one entry**;
  the entry is bounded by `nettyMaxFrameSizeBytes = maxMessageSize + MESSAGE_SIZE_FRAME_PADDING`
  (`5,253,120` bytes by default, scaling with `maxMessageSize`). Cursor-ledger entries are **not compressed**.
* **Metadata store / ZooKeeper** (`ManagedCursorInfo`) — used at cursor close, or when the range count is at or
  below `managedLedgerMaxUnackedRangesToPersistInMetadataStore`. The znode is bounded by `jute.maxbuffer`
  (10 MB default) and **can be compressed** via `managedCursorInfoCompressionType` (NONE by default; LZ4/ZSTD/…
  supported, since PIP-146).

**The serialization format (since [PR #9292](https://github.com/apache/pulsar/pull/9292)).** When
`managedLedgerPersistIndividualAckAsLongArray=true` (the default), the ranges are written via
`RoaringBitSet.toLongArray()` into a `repeated LongListMap { int64 key; repeated int64 values }` (one
`LongListMap` per ledger, `key` = ledgerId, `values` = the bitmap words). `RoaringBitSet.toLongArray()` is the
standard `java.util.BitSet.toLongArray()` contract: it returns a **dense** `long[]` of length
`highestSetBit/64 + 1`, i.e. **one 64-bit word for every 64 entry-ids up to the highest acknowledged entry in
that ledger, including all-zero interior/leading words**. The batch deletion `deleteSet` uses the same dense
`BitSet.toLongArray()` representation.

`RoaringBitmap` (the underlying data structure) supports a compact native serialization
(`RoaringBitmap.serialize` / `RoaringBitmap.deserialize`, sized via `serializedSizeInBytes()`) that stores
*containers* (array / bitmap / run) and is independent of the value span — but Pulsar currently discards that
compactness by going through `toLongArray()`.

# Motivation

**The persisted ack-state encoding is a dense bit array whose size is proportional to the backlog, not to the
number of acknowledgment holes.** Because `toLongArray()` emits one word per 64 entry-ids up to the highest
acknowledged entry-id per ledger, the byte cost scales with the **entry-id span** the cursor covers (i.e. the
backlog from the mark-delete position to the newest acknowledged message), regardless of how few actual ack
holes exist.

The tables below are produced by `ManagedCursorAckStateSizeTest` (see *General Notes*). Ack holes are spread
across the whole backlog with ~50 % Gaussian variance in the gap between holes (so they are not artificially
periodic). The headline table fixes the density at **~5 % of the backlog unacknowledged** and varies the
backlog:

| backlog | ack holes (~5 %) | dense raw (BK) | dense + LZ4 (ZK) | dense transient heap / flush | `RoaringBitmap.serialize` | Roaring + LZ4 |
|---:|---:|---:|---:|---:|---:|---:|
| 100 K | 5,000     | 0.02 MB | 0.01 MB | 0.01 MB | 0.02 MB | 0.01 MB |
| 1 M   | 50,000    | 0.16 MB | 0.09 MB | 0.12 MB | 0.16 MB | 0.06 MB |
| 10 M  | 500,000   | 1.63 MB | 0.85 MB | 1.19 MB | 1.57 MB | 0.64 MB |
| 100 M | 5,000,000 | **16.33 MB** | **8.52 MB** | **11.93 MB** | 15.67 MB | **6.38 MB** |

This has three consequences:

1. **BookKeeper cursor ledger — the major problem: no compression.** The cursor-ledger `PositionInfo` entry is
   written raw, so a 100 M-message backlog with ~5 % ack holes is a **~16.33 MB** entry — it exceeds the ~5 MB
   single-entry frame limit (`nettyMaxFrameSizeBytes`) and the add fails. The count cap
   `managedLedgerMaxUnackedRangesToPersist` does **not** prevent this: it caps **set bits (cardinality)**, not
   bytes, and the dense byte cost tracks the backlog regardless of the cap. This is why raising the persistence
   caps to reduce redelivery (e.g. the `200000` in [#23990](https://github.com/apache/pulsar/discussions/23990))
   is not byte-safe today.

2. **Metadata store (ZooKeeper) — compression helps, but far less than on idealized data.** With irregular
   (non-periodic) ack holes LZ4 only roughly halves the dense array at this density (**16.33 MB → 8.52 MB** at
   100 M / 5 %), which still exceeds the ~5 MB target and approaches the 10 MB `jute.maxbuffer` limit. (Perfectly
   periodic holes compress ~200×, but that is not representative of real workloads.) So compression mitigates
   the ZK path at low fragmentation but not at moderate-to-high fragmentation over a large backlog.

3. **Heap / GC on every flush, on both paths — which compression cannot fix.** Serialization first
   **materializes the entire dense `long[]`** (and deserialization rebuilds it): **~11.93 MB of transient
   garbage per flush** at a 100 M backlog, *regardless of hole count or LZ4*, churned on every mark-delete.
   `RoaringBitmap.serialize` / `deserialize` stream container-sized buffers and allocate a small amount that
   tracks the hole count, not the backlog.

**The dense size and heap track the backlog, not the ack holes.** At a fixed 100 M backlog the dense entry is
~16 MB at *every* hole density, while the proposed `RoaringBitmap.serialize` tracks the actual number of holes
(and is far smaller — even smaller than dense+LZ4 — at the low-to-moderate fragmentation typical of real
subscriptions):

| ack-hole density @ 100 M backlog | ack holes | dense raw | dense + LZ4 | `RoaringBitmap.serialize` | Roaring + LZ4 |
|---:|---:|---:|---:|---:|---:|
| 0.1 % | 100 K | 16.41 MB | 0.93 MB | 0.43 MB | **0.41 MB** |
| 1 %   | 1 M   | 16.39 MB | 5.02 MB | 3.86 MB | **3.87 MB** |
| 5 %   | 5 M   | 16.33 MB | 8.52 MB | 15.67 MB | **6.38 MB** |
| 10 %  | 10 M  | 16.26 MB | 10.13 MB | 15.67 MB | **8.32 MB** |

`RoaringBitmap.serialize` alone is far smaller at sparse/moderate densities (0.43 MB at 0.1 %) and converges
to the dense bitmap at ≥ 5 %, where it becomes essentially incompressible-as-stored. Combined with compression,
**Roaring + LZ4 is the smallest at *every* density** — and beats the current `dense + LZ4` everywhere (0.41 MB
vs 0.93 MB at 0.1 %; 6.38 MB vs 8.52 MB at 5 %) — because the run-optimized RoaringBitmap compresses better than
a raw dense bitmap. This is why the PIP proposes **both** the RoaringBitmap encoding (small at the common
densities, backlog-independent, and removing the dense-`long[]` materialization/heap at *every* density) **and**
cursor-ledger compression (which compounds with it and bounds the heavily-fragmented case — note that even
`Roaring + LZ4` is ~6.4 MB at the extreme 100 M / 5 % point, so the persistence count cap still matters).

**Backlog ceiling — the concrete current problem.** Because the dense size scales with the backlog and the
BookKeeper cursor ledger is **not compressed**, there is a hard ceiling on how large a subscription's backlog
can grow before its ack-state no longer fits the default ~5 MB cursor-ledger entry (`nettyMaxFrameSizeBytes`).
With the current dense encoding that ceiling is only **~30 million messages — regardless of how few ack holes
there are** (projected from the linear scaling above):

| ack-hole density | max backlog that fits a 5 MB BK entry (current dense encoding) |
|---:|---:|
| 0.1 % | ~30.5 M |
| 1 %   | ~30.6 M |
| 5 %   | ~30.7 M |
| 10 %  | ~30.8 M |

Once a subscription's backlog passes ~30 M messages, the cursor can no longer persist its full ack-state to a
BookKeeper entry: the add exceeds `nettyMaxFrameSizeBytes`, the persistence cap truncates ack ranges, and the
dropped acknowledgments are redelivered after broker restart / cursor recovery. The ceiling is essentially
**independent of the number of ack holes** — it is purely a function of the backlog — which is the clearest
illustration that the dense encoding charges for the backlog rather than for the acknowledgment state it
actually needs to record. The proposed `RoaringBitmap` encoding removes this backlog-bound ceiling (the
serialized size tracks the number of ack holes instead of the backlog span).

**Batch deletion indexes are a second, independent consumer of the same entry.** When
`acknowledgmentAtBatchIndexLevelEnabled=true` (the default), every partially-acknowledged *batched* entry is
tracked as a `BatchedEntryDeletionIndexInfo` (a per-batch `deleteSet` BitSet) serialized into the **same**
cursor-ledger entry as the ranges above, using the same dense `BitSet.toLongArray()` representation.

The backlog below is a **message count, not an entry count**: the topic stores `backlog / batchSize` *batched
entries* (one BK entry per batch), so a 1 M-message backlog is 100,000 entries at B=10 but only 10,000 at
B=100. Each tracked entry costs `position (~13 B) + deleteSet (ceil(batchSize/64) longs)`. BK storage
(uncompressed) for a **1 M-message backlog** — cell = size *(partially-acked batches tracked)*:

| ack density | B=10 | B=25 | B=50 | B=75 | B=100 |
|---:|---:|---:|---:|---:|---:|
| **batched entries** (= 1 M / B) | 100,000 | 40,000 | 20,000 | 13,334 | 10,000 |
| 0.1 % | 14 KB (988) | 15 KB (988) | 17 KB (988) | 17 KB (988) | 18 KB (988) |
| 1 %   | 146 KB (9,983) | 155 KB (9,983) | 168 KB (9,983) | 170 KB (9,714) | 163 KB (9,005) |
| 5 %   | 731 KB (50,030) | 622 KB (39,290) | 377 KB (20,000) | 292 KB (13,334) | 256 KB (10,000) |
| 10 %  | **1.32 MB (90,378)** | 661 KB (40,000) | 392 KB (20,000) | 314 KB (13,334) | 271 KB (10,000) |

The total is driven by *how many batches are partially acknowledged*, not by the per-entry size (a B=100 entry
is barely larger than a B=25 one — both are position-dominated, with a 1-long `deleteSet` for B ≤ 64). There
are two regimes:

* **Low fragmentation (0.1 %):** holes are sparse and each lands in a distinct batch, so the tracked count is
  ~constant (≈ 988) across batch sizes; the total then grows slightly with the per-entry `deleteSet` size, so
  **B=100 > B=25**.
* **Moderate-to-high fragmentation (≥ 5 %):** almost every batch has a hole, so the tracked count follows
  `backlog / batchSize` — B=25 has ≈ 4× more entries than B=100 (≈ 39 k vs 10 k), and that count dominates, so
  **B=25 > B=100** (e.g. 622 KB vs 256 KB at 5 %).

At a 10-message batch size and 10 % fragmentation this is **~1.3 MB on its own at only a 1 M backlog**, and it
*adds* to the individual-range bytes competing for the shared ~5 MB entry — it scales the same way and benefits
from the same RoaringBitmap encoding.

# Goals

## In Scope

* Persist `individualDeletedMessages` and the `BatchedEntryDeletionIndexInfo` `deleteSet` using
  **`RoaringBitmap.serialize`** (deserialized with `RoaringBitmap.deserialize`) instead of the dense
  `toLongArray()` representation. The persisted size becomes a function of the number/shape of ack holes and is
  **independent of the backlog span**; serialization/deserialization heap becomes small and bounded.
* Add **compression of BookKeeper cursor-ledger entries** (`PositionInfo`), reusing the same mechanism and
  configuration already used for `ManagedCursorInfo` in the metadata store, so the cursor-ledger path gets the
  same protection the ZK path already enjoys.
* Full backward/forward compatibility: brokers continue to read the existing dense `LongListMap` /
  `MessageRange` encodings, and the new encoding is gated so a cluster can be upgraded and rolled back safely.

## Out of Scope

* Changing the default values of `managedLedgerMaxUnackedRangesToPersist`,
  `managedLedgerMaxBatchDeletedIndexToPersist`, or
  `managedLedgerMaxUnackedRangesToPersistInMetadataStore`. Once the encoding is backlog-independent and the
  cursor ledger is compressed, raising these caps becomes safe and can be proposed separately.
* The in-memory representation of `individualDeletedMessages` / `batchDeletedIndexes` (already RoaringBitmap /
  BitSet backed). This PIP only changes how that state is **serialized** for persistence and transfer.
* The wire encoding of `ack_set` in the client protocol (`CommandMessage`/`CommandAck`). This PIP is about
  cursor *persistence*; a follow-up may apply the same idea to the protocol.

# High Level Design

Add a new, self-describing serialization for the per-ledger acknowledgment bitmaps and for batch deletion
`deleteSet`s that stores the RoaringBitmap's own compact serialized form instead of a dense `long[]`:

* On **write**, when the new format is enabled, each per-ledger bitmap is serialized with
  `RoaringBitmap.serialize(...)` into a new, distinct protobuf field; the batch `deleteSet` is likewise stored
  as a serialized RoaringBitmap.
* On **read**, the broker decodes whichever field is present — the existing dense `values`/`deleteSet` field
  (old data) or the new RoaringBitmap field (new data) — so recovery transparently handles both. This mirrors
  the self-describing approach already used for metadata compression (a magic header selects the codec).
* The **BookKeeper cursor-ledger entry** is compressed with the same codec abstraction
  (`CompressionCodecProvider`) and configuration already wired for `ManagedCursorInfo` in `MetaStoreImpl`,
  with the compression type recorded in the entry so reads auto-detect it.

The result: persisted ack-state size and serialization heap track the number of ack holes, not the backlog;
the cursor-ledger entry is bounded and small in the common case; and large, sparse, out-of-order ack patterns
no longer risk exceeding the BookKeeper frame limit.

# Detailed Design

## Design & Implementation Details

**Serialization (managed-ledger, `org.apache.bookkeeper.mledger.impl`).**

* `MLDataFormats.proto`: add a new representation for the per-ledger ranges that carries the RoaringBitmap
  serialized bytes, alongside the existing `LongListMap`. Two compatible options to be finalized during review:
  (a) a new `repeated LedgerRanges { int64 ledgerId = 1; bytes roaringBitmap = 2; }` field on `PositionInfo`
  and `ManagedCursorInfo`; or (b) add `optional bytes roaringBitmap` to `LongListMap` and leave `values` unset
  when the new format is used. Add an analogous `optional bytes roaringDeleteSet` to
  `BatchedEntryDeletionIndexInfo` (kept alongside the existing `repeated int64 deleteSet`). A reader prefers the
  RoaringBitmap field when present and falls back to the dense field otherwise.
* `ManagedCursorImpl`: `buildLongPropertiesMap` / the `persistPositionToLedger` and `persistPositionMetaStore`
  paths emit the RoaringBitmap field when the feature is enabled; `recoverIndividualDeletedMessages` and
  `recoverBatchDeletedIndexes` decode either field. `OpenLongPairRangeSet` gains direct access to the
  underlying `RoaringBitmap` for serialization, avoiding the `toLongArray()` round-trip entirely (eliminating
  the dense transient allocation).
* The encoders are first extracted into a small, side-effect-free helper so both production and the
  size-characterization test exercise the exact same logic.

**Cursor-ledger compression.**

* Extend the compression already implemented in `MetaStoreImpl` (`compressCursorInfo` /
  `compressManagedInfo`, using `CompressionCodecProvider` and a magic-header + `ManagedCursorInfoMetadata`)
  to the BookKeeper cursor-ledger write/read in `persistPositionToLedger` / cursor recovery. The compression
  type is recorded in the entry so reads auto-detect it and old (uncompressed) entries continue to be read.

**Why RoaringBitmap.serialize and not toLongArray.** `RoaringBitmap.serialize` writes containers (array
container for sparse, run container for runs, bitmap container only for genuinely dense regions), so its size
depends on the number/shape of set bits, not on the highest set index. `runOptimize()` makes the common
"holes every N messages" pattern collapse to run containers. `serializedSizeInBytes()` gives the exact size and
`deserialize` reconstructs in place without materializing a dense array.

## Public-facing Changes

### Configuration

* `managedCursorInfoCompressionType` (existing): its scope is extended/clarified to also cover BookKeeper
  cursor-ledger entries, **or** a new `managedCursorLedgerCompressionType` is introduced (to be decided in
  review) so the cursor-ledger codec can be configured independently. Default remains `NONE` for compatibility.
* A feature flag (e.g. `managedLedgerPersistAckStateAsRoaringBitmap`, default `false` initially) gates writing
  the new RoaringBitmap encoding, so the format can be enabled cluster-wide only after all brokers can read it.

### Binary protocol

No client-facing protocol change. The change is confined to cursor persistence (BookKeeper cursor ledger and
metadata-store `ManagedCursorInfo`) — internal `MLDataFormats` messages.

### Metrics

Reuse the existing truncation counters
(`pulsar.broker.managed_ledger.cursor.persist.unacked_ranges.truncated`,
`…batch_deleted_indexes.truncated`). Optionally add a gauge for the serialized cursor-state size to make the
improvement observable.

# Monitoring

Operators can watch the existing truncation metrics (they should drop to ~zero for previously-affected
subscriptions once the caps can safely be raised in a follow-up) and broker direct-memory / GC behavior
(reduced allocation churn on cursor flush). A serialized-size gauge, if added, can alert when a cursor
approaches the frame/znode budget.

# Security Considerations

None. The change does not add endpoints, protocol commands, or alter authorization. It only changes the
serialized representation of internal cursor metadata.

# Backward & Forward Compatibility

## Upgrade

No action required. Upgraded brokers read existing dense `LongListMap` / `MessageRange` / `deleteSet` data
unchanged. Writing the new RoaringBitmap encoding is gated by the feature flag (default off); enable it only
after all brokers in the cluster are upgraded to a version that can read it. Cursor-ledger compression is
self-describing (type recorded per entry), so uncompressed legacy entries continue to be read.

## Downgrade / Rollback

While the feature flag is off, persisted data is byte-identical to today, so rollback is unconditional. After
the flag has been enabled, a cursor may have RoaringBitmap-encoded and/or compressed entries that an older
broker cannot read; to roll back, disable the flag and let the next flush rewrite each cursor in the legacy
format (or document a minimum target version), exactly as with the existing `managedCursorInfoCompressionType`
rollout (PIP-146). This will be specified precisely in the implementation PR.

## Pulsar Geo-Replication Upgrade & Downgrade/Rollback Considerations

None. Cursor persistence is local to a cluster; geo-replication transfers message data over the binary
protocol and does not read another cluster's cursor metadata.

# Alternatives

* **Add a serialized-size guard only** (truncate/fall back before `nettyMaxFrameSizeBytes`). Bounds bytes but
  *discards* ack state under pressure (causing redelivery) instead of fixing the root-cause inefficiency.
* **Compress the cursor ledger but keep the dense `toLongArray()` encoding.** Compression helps, but on
  realistic (irregular) ack patterns LZ4 only roughly halves the dense array at moderate fragmentation
  (16.33 MB → 8.52 MB at 100 M / 5 %), and it does **not** fix the heap/GC cost — the full dense `long[]` is
  still materialized on every serialize/deserialize. RoaringBitmap shrinks the common sparse/moderate case far
  more and removes the dense materialization; compression then covers the heavily-fragmented case. The two
  changes are complementary, which is why this PIP proposes both.
* **Keep dense and raise the caps.** Not byte-safe, as shown in the Motivation.

# General Notes

The numbers in this PIP are produced by a reproducible characterization test,
`ManagedCursorAckStateSizeTest` (managed-ledger module), which builds the real RoaringBitmap-backed
`RangeSetWrapper` and generated `MLDataFormats` messages. For each representative backlog (10 K … 100 M
messages) and ack-hole density (~0.1 %, ~1 %, ~5 %, ~10 % unacked, with ~50 % Gaussian variance in the gap so
the holes are not artificially periodic) it measures the current dense `LongListMap` size (raw and LZ4), the
transient heap that serialization allocates, and the proposed `RoaringBitmap.serialize` size (raw and LZ4, with
`runOptimize()` applied before serialization). The code is pushed to the `lhotari/pulsar` fork branch backing
this proposal so the calculated values can be independently verified.
